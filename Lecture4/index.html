Почему не просто «тестирую»?

Как врач ставит диагноз

Пациент приходит с болью в животе. Врач не говорит «буду лечить» — он классифицирует: острая/тупая, локализованная/разлитая, связана с едой? От этого зависит метод лечения.

В тестировании то же самое. Классификация позволяет выбрать инструменты, распределить усилия и говорить с командой на одном профессиональном языке.

Основные срезы

Классификация тестирования

По уровням

  • Модульное (компонентное) — отдельные кирпичи
  • Интеграционное — связи между модулями
  • Системное — вся система целиком
  • Приёмочное — готовность к эксплуатации

По запуску кода

  • Статическое (без выполнения: код-ревью, SonarQube, документация)
  • Динамическое (выполнение программы)

По знанию устройства

  • Белый ящик (видим код — unit-тесты)
  • Чёрный ящик (только ввод/вывод — UI-тесты)
  • Серый ящик (частичный доступ: БД, логи, API)

По автоматизации

  • Ручное (исследовательское, юзабилити)
  • Автоматизированное (регресс, нагрузка, повторяемость)

По целям (самое важное)

Функциональное

Проверка функций: регистрация, оплата, добавление товара…

Нефункциональное

Производительность, безопасность, usability, совместимость, установка…

Золотая середина

Интеграционное тестирование

Модули по отдельности работают, а вместе — нет. Ошибки на стыках: форматы данных, последовательность вызовов, права доступа.

Кирпичи качественные, но электрики забыли соединить провода — интеграционное тестирование проверяет соединения.

Стратегии интеграции

Big Bang

Собрали всё вместе и тестируем. Минус: сложно локализовать ошибку.

Top-Down

Сверху вниз. Для имитации отсутствующих модулей — стабы (stubs).

Bottom-Up

Снизу вверх. Для вызова тестируемых модулей — драйверы (drivers).

Главный инструмент

Postman: тестирование API

Отправка запросов

GET, POST, PUT, DELETE, заголовки, тело (JSON/XML).

Коллекции

Организация запросов по модулям.

Автотесты на JS

Проверка статус-кодов, тела ответа, времени.

Окружения (Environments)

Переменные для разных серверов: dev, staging, prod.

Newman

Запуск коллекций из CLI, интеграция в CI/CD.

Пример тестов в Postman (вкладка Tests)

pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

pm.test("Response contains products", function () {
    var jsonData = pm.response.json();
    pm.expect(jsonData).to.be.an('array');
    pm.expect(jsonData.length).to.be.greaterThan(0);
});

pm.test("Response time < 500ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(500);
});

Сценарии: первый запрос создаёт ресурс → из ответа извлекаем ID → сохраняем в переменную → второй запрос использует ID.

API — любимая цель атак

Тестирование безопасности через API

Нарушение прав доступа

Попробуйте запросить данные другого пользователя, изменив ID в URL.

Отсутствие rate limit

Тысяча запросов в секунду — DoS-уязвимость.

Передача данных в открытом виде

Нет HTTPS / чувствительные поля в открытом тексте.

Негативные сценарии

Отрицательные цены, буквы вместо цифр, XSS в полях.

Где находится API-тестирование в нашей классификации?

  • Уровень: интеграционное (проверка связи клиент-сервер или сервер-БД)
  • Знание устройства: серый ящик (знаем эндпоинты, но не код)
  • Автоматизация: может быть ручным (Postman вручную) и автоматизированным (коллекции + Newman)
  • Цели: чаще функциональное, но также нефункциональное (нагрузка, безопасность)

Ваше первое API-тестирование

Теория

Повторить HTTP-методы, статус-коды, структуру JSON, REST.

Практика

Установить Postman (desktop). Создать коллекцию с тремя запросами к любому открытому API (GitHub, погода и т.п.). Написать для каждого минимум два теста (статус-код и структура). Прислать скриншот с успешным прогоном.
